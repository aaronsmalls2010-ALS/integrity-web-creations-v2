# IWC Admin CRM + Invoicing ‚Äî Design Spec

**Date:** 2026-06-07
**Project:** `integrity-web-creations-v2` (Astro 6 SSR, Vercel, Tailwind 4)
**Owner / sole user:** Integrity Web Creations (single-admin system)
**Status:** Approved design ‚Äî ready for implementation planning

---

## 1. Purpose & Goal

A single-user admin CRM + invoicing system living inside the existing IWC v2 Astro site, behind a secure admin login. It manages clients, generates and sends branded invoices, collects payment (Stripe + manual), tracks payment history, automates recurring billing (hosting/retainers), and produces simple income/aging reports with CSV export for the accountant.

**Explicit non-goal:** This is **not** a full accounting system. No expense ledger, no double-entry, no bank reconciliation, no P&L beyond income reporting. If those needs ever arise, buy QuickBooks/Wave instead of extending this.

---

## 2. Locked Decisions

| Area | Decision |
|------|----------|
| Accounting depth | Invoicing + payment tracking only |
| Payments | Stripe hosted "Pay Now" **and** manual mark-paid (check/Zelle/cash) |
| Invoice delivery | Email with secure hosted-page link **+ attached PDF** |
| Architecture | Protected `/admin/*` routes in v2 Astro + Supabase (Postgres + Auth + RLS) + Stripe + Vercel Cron |
| Templates | Saved line-item presets, one branded invoice layout, email message templates, duplicate-invoice |
| Recurring billing | Auto-generate per schedule; per-schedule toggle: auto-send vs. leave-as-draft |
| Admin auth | Supabase Auth, email + password + TOTP 2FA, single allowlisted user, public signup disabled |
| Email delivery | **Postmark** transactional ESP; sent from branded `billing@integritywebcreations.com` with full domain authentication (SPF/DKIM/DMARC). Bluehost SMTP retained for the public contact form only. |
| Data import | Start fresh ‚Äî no migration tooling |
| Refunds | Out of v1 ‚Äî handled directly in Stripe, recorded manually if needed later |
| Owner alerts | Email the owner when an invoice is **viewed** and when it is **paid** |
| Currency | USD only |
| Reporting basis | Cash basis (income recognized on payment date) |
| Timezone | America/New_York for all date logic |

## 3. Architecture Overview

- **App:** New `/admin/*` route group in `integrity-web-creations-v2` (Astro SSR via `@astrojs/vercel`). One public client-facing route: `/i/[token]`.
- **Data + Auth:** Supabase (Postgres, Auth, Row Level Security, Storage for logo + cached PDFs).
- **Payments:** Stripe ‚Äî **hosted Checkout Sessions** (card data never touches our server ‚Üí PCI SAQ A) + webhook for payment confirmation.
- **Email:** Postmark API for all transactional mail (invoice send, reminders, receipts, owner alerts).
- **Scheduled jobs:** Vercel Cron ‚Äî daily recurring-invoice generation + overdue sweep, secured by a shared secret header.
- **Secrets:** All server-side only (Supabase service-role key, Stripe secret, Postmark token, cron secret). Never exposed via `PUBLIC_` env or client bundles.

### Money representation
All monetary values stored as **integer cents** (or Postgres `numeric(12,2)`), never floating point. All arithmetic (subtotal, tax, discount, totals, balances) done in cents.

## 4. Data Model (Supabase / Postgres)

> All tables have `id` (uuid), `created_at`, `updated_at`. RLS locks every table to the authenticated admin (see Security). Nothing financial is ever hard-deleted.

### `clients`
- `name`, `business_name`, `email`, `phone`
- `address_line1`, `address_line2`, `city`, `state`, `postal_code`, `country`
- `notes`
- `stripe_customer_id` (nullable)
- `status` (`active` | `archived`) ‚Äî soft archive, never hard-delete if invoices exist

### `services` (saved line-item presets)
- `name`, `description`
- `default_unit_price` (cents), `default_quantity`
- `taxable` (bool)
- `active` (bool)

### `invoices`
- `invoice_number` (text, sequential w/ prefix; **allocated on send**, gap-free, never reused)
- `client_id` (FK)
- **Snapshot fields (captured at send time, immutable thereafter):**
  - `issuer_snapshot` (jsonb: business name, address, logo, payment instructions as they were)
  - `bill_to_snapshot` (jsonb: client name/business/address/email as they were)
- `status` (`draft` | `sent` | `viewed` | `paid` | `partial` | `overdue` | `void`)
- `issue_date`, `due_date`
- `currency` (`USD`)
- `subtotal_cents`, `tax_rate`, `tax_cents`, `discount_cents`, `total_cents`
- `amount_paid_cents`, `balance_cents`
- `terms` / `notes`
- `public_token` (high-entropy, unguessable; used for `/i/[token]`)
- `stripe_checkout_session_id`, `stripe_payment_intent_id` (nullable)
- `pdf_storage_path` (nullable; cached in Supabase Storage)
- `sent_at`, `viewed_at`, `paid_at`
- `recurring_schedule_id` (nullable FK ‚Äî set if generated by automation)

**Immutability rule:** once `status != draft`, financial fields and line items are locked. Edits to a sent invoice are not allowed ‚Äî void and reissue instead. Only drafts are deletable; sent/paid invoices can only be **voided** (number retained).

### `invoice_line_items`
- `invoice_id` (FK)
- `service_id` (nullable FK ‚Äî for reference only; description is snapshotted)
- `description`, `quantity`, `unit_price_cents`, `amount_cents`, `taxable` (bool), `sort_order`

### `payments`
- `invoice_id` (FK)
- `amount_cents`
- `method` (`stripe` | `check` | `zelle` | `cash` | `other`)
- `reference` (check #, Zelle confirmation, etc.)
- `stripe_payment_intent_id` (nullable)
- `paid_at`
- `note`
- Supports **partial payments** (multiple rows per invoice; invoice `amount_paid`/`balance` recomputed, status ‚Üí `partial` then `paid` at zero balance)

### `recurring_schedules`
- `client_id` (FK)
- `name` (e.g. "Monthly Hosting")
- `line_items` (jsonb ‚Äî template of items to generate)
- `interval` (`monthly` | `quarterly` | `annual`), `interval_count`
- `next_run_date`
- `auto_send` (bool ‚Äî true = generate + send; false = generate as draft)
- `active` (bool)
- `last_generated_period` (guard for idempotent generation ‚Äî no duplicate invoice per period)
- `last_invoice_id` (nullable)

### `email_templates`
- `key` (`invoice_send` | `reminder` | `receipt` | `owner_alert`)
- `subject`, `body` (supports placeholders: `{{client_name}}`, `{{invoice_number}}`, `{{amount}}`, `{{due_date}}`, `{{pay_link}}`, etc.)

### `settings` (single row)
- Business: `business_name`, address fields, `logo_storage_path`, `reply_to`, `from_email`
- Invoice defaults: `invoice_number_prefix`, `next_invoice_number` (atomic counter), `default_due_days`, `default_tax_rate`, `default_terms`, `payment_instructions`
- Integration status flags (Stripe connected, Postmark connected)

### `activity_log`
- `entity_type` (`invoice` | `payment` | `client` | ...), `entity_id`
- `action` (`created` | `sent` | `viewed` | `paid` | `payment_recorded` | `voided` | ...)
- `detail` (jsonb), `occurred_at`

### `webhook_events` (idempotency)
- `provider` (`stripe`), `event_id` (unique), `processed_at` ‚Äî dedupe guard so a re-sent webhook never double-records.

## 5. Core Flows

### 5.1 Invoice lifecycle
1. **Create draft** ‚Äî pick client, add line items (from `services` presets or custom), set issue/due dates, tax rate, discount. No number yet.
2. **Send** ‚Äî atomically allocate next `invoice_number`; snapshot issuer + bill-to details; render + cache PDF; create Stripe Checkout Session (Pay-Now link) with `invoice_id` in metadata; email client via Postmark (hosted-page link + PDF attachment); set `status = sent`, `sent_at`.
3. **Client views** `/i/[token]` ‚Üí `status = viewed` (if previously sent), `viewed_at`, **owner alerted**.
4. **Payment**
   - *Stripe:* client pays via hosted Checkout ‚Üí webhook (idempotent, signature-verified) records a `payment` row, recomputes balance, sets `paid`/`partial`, emails receipt, **owner alerted**.
   - *Manual:* owner records check/Zelle/cash payment ‚Üí same recompute path.
5. **Duplicate** ‚Äî clone any invoice (line items + client) into a new draft.
6. **Void** ‚Äî sent/paid invoices flip to `void` (number retained); drafts can be deleted.

### 5.2 Recurring billing (daily Vercel Cron)
- Find `recurring_schedules` where `active` and `next_run_date <= today` and `last_generated_period != current period`.
- Generate invoice from `line_items`; if `auto_send` ‚Üí run the Send flow; else leave as draft.
- Set `last_generated_period`, advance `next_run_date`. Idempotent ‚Äî a double cron run cannot create two invoices for the same period.

### 5.3 Overdue sweep (daily Vercel Cron)
- Mark `sent`/`viewed`/`partial` invoices past `due_date` as `overdue`.
- Optionally send a reminder email (using the `reminder` template) per a configurable cadence.

### 5.4 Auth
- Supabase email/password + TOTP 2FA. Public signup disabled; only the owner's allowlisted email can authenticate. RLS enforces that only the authenticated owner reads/writes any table.

## 6. Admin Screens (`/admin/*`)

- **Login** (`/admin/login`) ‚Äî email + password + TOTP, rate-limited.
- **Dashboard** ‚Äî outstanding total, overdue count/amount, paid-this-month, recent activity, quick actions.
- **Clients** ‚Äî list/search; detail = contact info, invoice history, payment history, running balance.
- **Invoices** ‚Äî list with status filters; new; detail (view/edit-draft/send/record-payment/duplicate/void/download-PDF).
- **Recurring** ‚Äî schedules list + editor (interval, line items, auto-send toggle, next run).
- **Services** ‚Äî manage line-item presets.
- **Reports** ‚Äî monthly income (cash basis), outstanding/aging, **CSV export** for accountant.
- **Settings** ‚Äî business info + branding (logo upload), invoice defaults, email templates, integration status (Stripe/Postmark).

---

## 7. Client-Facing Page (`/i/[token]`)

- Public but unguessable (`public_token`); `noindex`.
- Branded invoice view: line items, totals, payment instructions, **Pay Now** (Stripe Checkout), **Download PDF**.
- Reads a **single** invoice via a server-only API route using the service role ‚Äî the `invoices` table is never directly exposed to the client. Logs `viewed`.

---

## 8. Security Model (non-negotiable)

- **RLS** on every table, scoped to the authenticated owner. Service-role key server-side only.
- **2FA** (TOTP) on admin login; rate-limit login + token + cron endpoints; sensible session length.
- **Public signup disabled**, owner email allowlisted.
- **Stripe:** hosted Checkout (SAQ A ‚Äî no card data on our servers); **webhook signature verification**; **idempotent** event processing via `webhook_events`.
- **Cron endpoints** protected by a shared secret header; idempotent generation.
- **CSRF protection** on all admin mutations; input validation + sanitization everywhere.
- **Secrets** in env only; nothing sensitive in client bundles; no `PUBLIC_` leakage.
- **Email auth:** SPF + DKIM + DMARC configured for the sending domain (Postmark).
- **Security headers** (CSP, etc.); SRI for any external scripts; `noindex` on `/i/*` and all `/admin/*`.
- **Backups/export:** scheduled data export so financial records are recoverable independent of the DB host.
- Per project standard: full deploy security checklist before launch.

## 9. Known Technical Risk: PDF generation on serverless

Headless-Chrome PDF rendering is heavy on Vercel serverless. Plan:
- Build the invoice as **one HTML template**, reused for both the hosted page and the PDF.
- Render to PDF with a serverless-compatible approach; **cache the result to Supabase Storage** (`pdf_storage_path`) so we don't regenerate on every request.
- **Validate this early (Phase 2)** with a spike before building the full send flow. Fallback options if Chrome-on-serverless proves painful: a lightweight HTML-to-PDF library, or an external render step.

## 10. Out of Scope (YAGNI)

Expense tracking ¬∑ double-entry/books ¬∑ bank reconciliation ¬∑ multi-user/roles ¬∑ multi-currency ¬∑ time tracking ¬∑ project management ¬∑ estimates/quotes (candidate for a later phase) ¬∑ tax filing ¬∑ dunning/automated retries ¬∑ late-fee automation ¬∑ refund flow inside the app (handled in Stripe for v1).

## 11. Standards to Apply (per project memory)

- **Umami analytics** from day one (admin usage + client-facing page).
- **SEO/accessibility** standard where applicable (admin is `noindex`; semantic, accessible UI throughout).
- **No emojis** in UI ‚Äî lucide-react / SVG icons only.
- **Design excellence** ‚Äî portfolio-worthy, corporate-grade admin UI.
- **Mobile nav standard** for any responsive navigation.

## 12. Build Phases (vertical slices)

- **Phase 0 ‚Äî Foundation:** Supabase project, full schema + RLS, Auth + 2FA, admin shell + login, env/secrets, dev vs. prod environment separation + Stripe test mode.
- **Phase 1 ‚Äî Records:** Clients CRUD, Services presets, Settings + branding (logo upload).
- **Phase 2 ‚Äî Invoices (core):** create/edit drafts, send-time numbering, line items, PDF spike + hosted `/i/[token]` page, manual mark-paid, partial payments, duplicate/void. ‚Üê *first usable milestone*
- **Phase 3 ‚Äî Sending + Stripe:** Postmark integration + domain auth + email templates; Stripe Checkout Pay-Now + idempotent webhook + receipts + owner alerts.
- **Phase 4 ‚Äî Automation:** recurring schedules + daily generation cron; overdue sweep + reminders.
- **Phase 5 ‚Äî Insight:** dashboard + reports + CSV export.
- **Phase 6 ‚Äî Hardening + launch:** security review, rate limits, activity log polish, backups/export, Umami, full pre-launch checklist.
